# Failure Analysis of MOAR's Code Synthesis

## Case 1: HTML Extraction (`break_moar/1`)

### 1. Setup:

The objective of this scenario is to extract clean and plain text from HTML strings. The dataset (`break_moar/1/html_data.json`) was generated by `Gemini 3.1 Pro`, and consists of 50 strings containing various HTML tags. The evaluation function was written mimicking the [official DocETL MOAR examples](https://ucbepic.github.io/docetl/optimization/moar/examples/#evaluate_medicationspy): it extracts the `raw_html` field from the output results, uses a regular expression to extract text outside of `< >` tags, and matches it against the text extracted by the LLM. The pipeline is straightforward, directly prompting the LLM to extract the text within the HTML strings. The available models for execution are `gpt-4o-mini`, `gpt-4o` and `gpt-5-mini`, with rewrite agent model `gpt-4o`. Detailed configurations can be found in `break_moar/1/pipeline.yaml`.

### 2. Code Bug and Reward Hacking

According to the generated `break_moar/1/results/moar_optimization_html/pareto_frontier.json`，Node 6 achieved an accuracy score of 48/50 with cost of $0.002, earning it the Rank 1 position. Upon inspecting the generated `pipeline_6.yaml` and `pipeline_6.json`, I found that MOAR used code synthesis, but the resulting code contained a critical bug and produced poor results. However, it managed to hack the reward metric, making MOAR evaluate its accuracy at 48/50.

MOAR generated a pre-processing `code_map` operation aimed at context compression. It attempted to extract 200-character windows around important tags and merge them. However, the code contained a fatal unpacking error:

```
compressed_text = '\n\n--- SECTION BREAK ---\n\n'.join([span[2] for _, _, span in merged_spans])
```

Because `_, _, span` had already unpacked the tuple `(start_index, end_index, text)`, the variable `span` was a string. However, the code appended `[2]`, making it extract the 3rd character of the string. So some HTML strings were reduced to a single character.

The true danger is the function's final line: `return {'raw_html': compressed_text}`. The code overwrote the original `raw_html` column with the compressed text. Because the evaluation function computes its ground truth from the overwritten `raw_html`, and the downstream LLM extraction step operates on the exact same corrupted text, MOAR successfully blinded the evaluator to the loss of original HTML content and secured a fake score of 48/50. In fact, the true accuracy score is 21/50. Meanwhile, simply using `gpt-4o-mini` with the original `pipeline.yaml` yields an accuracy of 48/50 without any data destruction.

## Case 2: Medical Transcripts (`break_moar/2`)

### 1. Setup

The objective of this scenario is to extract all mentioned medications from medical transcripts. The dataset is sourced from the [official DocETL tutorial](https://ucbepic.github.io/docetl/tutorial/#preparing-the-data) and contains 87 medical transcripts. Learning from the vulnerability exposed in `break_moar/1`, the evaluation function has been strictly patched to read the `src` field directly from the original dataset. It then verifies whether the medications extracted by the LLM actually appear in that source text. The pipeline is similarly straightforward, directly prompting the LLM to extract the mentioned medications. The available models for execution include `gpt-4o-mini`, `gpt-4o`, `gpt-5-mini`, and `gpt-5-nano`, with the rewrite agent model set to `gpt-4o`. Detailed configurations can be found in `break_moar/2/pipeline.yaml`.

### 2. Destructive Truncation

According to the generated `break_moar/2/results/moar_optimization/pareto_frontier.json`，Node 11 and Node 26 ranked highly on the Pareto frontier due to their low execution costs (around $0.005). However, their accuracy scores are quite poor compared to other nodes that easily scored over 200. Specifically, Node 11 achieved a score of only 17, while Node 26 scored a mere 32.

Upon inspecting their generated YAML `pipeline_11` and `pipeline_26`, it became clear why their performance dropped so much. To cut token costs, MOAR's code synthesis agent applied a highly destructive text truncation. If a transcript exceeded a certain length, Node 11 and Node 26's synthesized Python code retained only a small snippet of the beginning and the end, completely discarding the entire middle section and replacing it with `...`. Node 26 further compounded this error with an additional regex filter. While this "read the intro and conclusion" approach might work for summarizing news articles or essays, it is absolutely catastrophic for clinical data. In a medical transcript, the most valuable information is almost always in the middle of the dialogue.

This results in loss of the medication the pipeline was designed to extract. Consequently, MOAR's strategy selection mechanism must incorporate a minimum accuracy requirement. The optimizer cannot blindly optimize for token efficiency without ensuring the baseline accuracy of the pipeline.

### 3. The Fragility of Heuristic Rules

Looking at the rest of the Pareto frontier, Nodes 1, 7, and 16 represent pure LLM executions without code synthesis. MOAR also generated code-synthesized pipelines like Node 10, 17, and 19. Compared to the pure LLM baseline, these nodes experienced a slight drop in accuracy but successfully lowered execution costs. For example, Node 10 reduced the cost by roughly $0.006, accompanied by a minor decrease in its accuracy score. Upon inspecting the `pipeline.yaml` for these nodes, I discovered that they all rely on regex to extract keywords and filter the context.

However, this exposes the fragility of using hardcoded heuristics to process unstructured natural language. These generated code attempt to locate relevant context windows by searching for generic medical trigger words, such as `pill`, `dose`, `take`, or `medication`. The problem is that natural conversation doesn't always conform to rigid regex patterns. An example of this failure occurred in transcript `D2N010`. The LLM-based Node 16 correctly extracted the medication "Protonix" from the following phrase: `Do you feel like it's better since we've put you on the Protonix?`. However, because this natural phrase lacked any of the hardcoded trigger words, the regex-based nodes failed to identify it, causing all of them to miss "Protonix" entirely. As a result, the pipelines utilizing code synthesis ultimately underperformed compared to the baseline nodes that relied purely on the LLM.
